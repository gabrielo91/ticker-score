# Learnings — DarkScore Project

What we've learned about spec-driven development, AI agent orchestration, and architecture governance while building the DarkScore risk score platform.

---

## On Spec-Kit and Spec-Driven Development

### What worked
- **The `.specify/` convention is powerful**. Having a single well-known directory for all governance documents means any AI agent (Claude, Copilot, Cursor, custom agents) can bootstrap itself without human instruction. It's the equivalent of `.github/` for specs.
- **Constitution as immutable rules** (not guidelines) gives agents hard boundaries. The distinction matters — guidelines get bent, rules get enforced.
- **Living plan.md (C13) is essential**. Our plan.md went stale within hours of creation. Adding the Living Plan Protocol (C13) means any cold-start agent reads 5 lines and knows exactly where we are and what to do next. The handoff instruction is just: "Read plan.md, continue from Current State."
- **Spec-first workflow prevents scope creep**. Every feature requires a spec amendment before implementation. This stopped agents from inventing requirements.

### What could be improved
- **spec-kit lacks built-in agent bootstrapping**. We had to invent AGENTS.md and the context chain (AGENTS.md → constitution → package CONSTITUTION.md → task). This should be part of the spec-kit convention.
- **No template for living plan**. spec-kit provides a plan.md template but it's static. A living plan template with Current State / Completed / Active / Upcoming sections should be standard.
- **Task management is split**. Tasks live in workspace notes (Intent) AND plan.md. We resolved this by making `.specify/` canonical and the workspace spec note for "live coordination." This works but requires discipline.
- **Constitution versioning is implicit**. When we went from C1-C11 → C12 → C13, we had to manually grep and update references everywhere. A version header or changelog in the constitution would help.

## Spec-Kit vs DarkScore Additions

### What spec-kit provides (out of the box)
| Convention | Purpose |
|-----------|---------|
| `.specify/memory/constitution.md` | Immutable project rules, loaded by memory-loader before any agent command |
| `.specify/specs/{feature}/spec.md` | Feature specification — what to build |
| `.specify/specs/{feature}/plan.md` | Implementation plan — how to build it |
| `.specify/specs/{feature}/tasks.md` | Task breakdown (we use plan.md for this instead) |
| `CLAUDE.md` / agent config files | Generated by `specify init` for specific AI tools |
| Memory-loader pattern | Constitution auto-loads into agent context before commands |
| Spec → Plan → Task flow | Structured progression from requirements to implementation |

### What we added on top (DarkScore innovations)
| Addition | Why spec-kit doesn't cover it | Overlap risk |
|----------|------------------------------|--------------|
| **AGENTS.md** (root entry point) | spec-kit generates tool-specific files (CLAUDE.md, .cursorrules). We needed a single universal entry point that works with ANY AI agent or tool — not tied to Claude or Cursor. | None — AGENTS.md is agent-agnostic; CLAUDE.md would be a symlink/subset if needed. |
| **Per-package CONSTITUTION.md** | spec-kit has one constitution at project level. We added scoped preambles per package so agents get local context without reading the full constitution. | Low — package files are 25-line summaries with "parent context" links, not rule copies. |
| **Automated boundary checker** (scripts/check-boundaries.ts) | spec-kit is a documentation framework, not an enforcement framework. We added build-time scripts that verify constitution rules programmatically. | None — this is enforcement, not documentation. |
| **No-any checker** (scripts/check-no-any.ts) | Same as above — spec-kit defines rules, we enforce them. | None. |
| **C4 diagrams as constraints (C11)** | spec-kit doesn't prescribe architecture diagrams. We added L1/L2/L3 Mermaid diagrams in `.specify/` and made them normative (violations = spec breach). | None — extends spec-kit's file structure naturally. |
| **Living Plan Protocol (C13)** | spec-kit's plan.md is a static planning artifact. We made it a living document with Current State / handoff protocol that gets updated after every wave. | None — same file, different lifecycle practice. |
| **Context chain (C8 enhancement)** | spec-kit assumes agents read the constitution. We defined the explicit reading ORDER: AGENTS.md → package CONSTITUTION.md → task → files. Prevents full-codebase exploration. | None — augments spec-kit's memory-loader pattern. |
| **Frontend guidelines (C12)** | spec-kit is stack-agnostic. We added React-specific Presentation-Domain-Data layering rules based on Martin Fowler's patterns. | None — domain-specific extension. |
| **Turbo validate pipeline** | spec-kit doesn't manage build systems. We wired constitution rules into Turborepo's task graph so `turbo validate` = "prove you followed the rules." | None — build tooling, not spec tooling. |
| **Breadcrumb links** | spec-kit files don't cross-reference each other. We added parent-context links so every document points back up to AGENTS.md and the constitution. | None — navigation aid only. |

### Design principle: Extend, don't overlap
Every addition follows one rule: **spec-kit owns the document structure, we own the enforcement and agent ergonomics**. We never created parallel files that duplicate spec-kit's conventions — we added files that spec-kit doesn't have a convention for (AGENTS.md, package CONSTITUTION.md, scripts/) and enhanced files that spec-kit defines but doesn't prescribe lifecycle for (plan.md → living document).

The only file that could theoretically overlap is AGENTS.md vs CLAUDE.md. We chose AGENTS.md because it's tool-agnostic. If a project uses Claude Code, CLAUDE.md can simply say `# Read AGENTS.md` — one line, zero duplication.

## On C4 Diagrams as Architecture Constraints

### Decision: Add L1, L2, and selective L3 diagrams. Skip L4.

### What worked
- **L2 Container diagram IS the harness**. The package dependency graph as a diagram gives agents a visual + parseable constraint. It directly reinforces Constitution C4 (Package Boundaries).
- **L3 only for complex packages** (data-providers, scoring-engine). These have internal pluggable architecture (registry pattern, strategy pattern) that benefits from visual clarity. Simple packages (cache, types, db) don't need L3.
- **Interactive diagram states** (walkthrough steps) help explain "how to add a new provider" or "how scoring flows" — useful for onboarding new agents or humans.
- **Mermaid format in .specify/ is portable**. Any Markdown renderer can display them.

### What could be improved
- **L4 (Code level) is actively harmful for AI agents**. It becomes stale instantly and gives agents false confidence about implementation details. Code IS the diagram at that level.
- **Diagram sync is manual**. The C4 diagrams exist in Intent notes (interactive JSON format) and .specify/ (Mermaid). If one changes, the other doesn't update automatically.

### Recommendation
C4 diagrams are worth it if:
1. You stop at L2 for simple packages, L3 for complex ones
2. You use them as constraints (C11), not just documentation
3. You keep them in a text format (Mermaid) that lives with the code

## On Agent Harness Design (Anti-Hallucination)

### Decision: Multi-layered enforcement with automated checking

### What worked
- **AGENTS.md as the single entry point**. Under 200 lines. Any agent, any tool. This is the most impactful file in the repo for AI collaboration.
- **Per-package CONSTITUTION.md** gives agents scoped context. An agent working on scoring-engine reads 25 lines and knows: "I'm pure computation. No I/O. No fetch. No database. Types only." This prevents the most common hallucination: importing things that shouldn't be imported.
- **Automated boundary checking** (scripts/check-boundaries.ts) catches violations at build time. This is the safety net — even if an agent ignores the docs, the script catches the import.
- **No-any checker** prevents the most common TypeScript escape hatch.
- **Context chain (C8)** — the explicit reading order (AGENTS.md → package CONSTITUTION.md → task note → only the files you'll modify) prevents agents from exploring the full codebase and getting confused.
- **"Stop and ask, don't guess"** rule is critical. Agents that guess invent requirements.

### What could be improved
- **ESLint import restrictions per package** would give real-time IDE feedback, but the scaffold complexity wasn't worth it in Phase 0. Add in Phase 1.
- **The boundary checker could be a pre-commit hook**. Right now it runs via turbo validate, but a git hook would catch violations even earlier.
- **Verification is trust-based**. We tell agents "run turbo validate before reporting done" but can't enforce it. A CI gate would be the real enforcement.

### Key insight
The harness works in layers, each catching errors the previous one missed:
1. **Context** (AGENTS.md, CONSTITUTION.md) — prevents errors before they happen
2. **IDE** (TypeScript strict mode) — catches errors as you type
3. **Build** (boundary checker, no-any checker) — catches errors at build time
4. **Submission** (pre-submission checklist) — last gate before code lands

## On Frontend Architecture (C12)

### Decision: Adopt Presentation-Domain-Data layering from Martin Fowler/Juntao Qiu

### Rationale
- React is a view library, not an architecture. Without explicit layering, AI agents will put fetch calls, business logic, and rendering in the same component.
- The four-layer model (Views → Hooks → Domain Models → Gateways) maps naturally to our package structure: Views = apps/web/components, Models = @darkscore/types + scoring-engine, Gateways = @darkscore/data-providers.
- Making it a constitution rule (C12) means it's not optional — every frontend change must respect the layers.

### Key learning
Frontend architecture rules are MORE important for AI agents than for human developers. Humans intuit separation of concerns. AI agents will happily put a fetch call inside a component if you don't explicitly forbid it.

## On Monorepo Tooling (Turborepo)

### Decision: Turborepo with pnpm workspaces

### What worked
- **Well-known tool = less hallucination**. AI models know Turborepo well. We didn't have to fight incorrect configurations.
- **Pipeline-based task graph** (turbo.json) naturally enforces build order based on package dependencies.
- **Adding custom pipeline tasks** (check:boundaries, check:no-any, validate) was straightforward.

### What could be improved
- **Root-only tasks need //#prefix syntax** which is non-obvious. Our harness scripts needed this pattern and it took a cycle to get right.

## On Documentation Deduplication

### Problem we hit
The same information (constitution rules, dependency table, architecture) appeared in 3+ places: spec note, .specify/ files, AGENTS.md, package CONSTITUTIONs.

### Resolution
- **Single canonical source** per document type (`.specify/` files are authoritative)
- **Appropriate summaries** at each level (AGENTS.md has the quick-ref table, packages have 25-line preambles)
- **Conflict resolution rule**: "L2 diagram wins" for dependency questions
- **Source of Truth table** in the spec note explicitly maps what lives where

### Key insight
Some duplication is acceptable and even desirable — agents shouldn't have to read 5 files to find one rule. The key is establishing which copy wins on conflict and keeping summaries short enough that they rarely drift.

## On Living Plans and Handoff Readiness

### Problem we hit
Plan.md was written once during W1-3 and was stale by the time W1-4 completed. A new agent would have found all tasks marked "Not Started" despite 4 being done.

### Resolution
- **C13 — Living Plan Protocol**: plan.md must be updated after every wave
- **Current State section**: always the first thing, always under 5 lines, always shows what's done and what's next
- **Cold-start instruction**: "Read plan.md, continue from Current State." That's the complete handoff.

### Key insight
The plan is not a planning artifact — it's a **program counter**. It tells the system where execution left off. Treating it as a living, always-current document is what makes multi-agent and multi-session development possible.
